Why this exists

A workflow for AI-agent work that has to be correct.

Most agent failures are not model failures. They are routing failures: vague goals, hidden risk, missing evidence, no reviewer, and no stop condition. ResearchLoop V4 makes those failure modes visible before code changes.

1

Track first

Light, standard, and heavy routes keep easy tasks fast while forcing risky work into full evidence and review.

2

Evidence-led

Assumptions, tests, diffs, blockers, and reviewer notes become explicit artifacts instead of chat history.

3

Human gates

G1, G2, and G3 separate problem approval, execution authorization, and final acceptance.

4

No silent drift

Plans, worktree checks, and retrospectives keep agents from changing scope when they encounter difficulty.

普通 Agent Prompt

Fast but fragile

  • Starts implementing before the problem is stable.
  • Mixes assumptions, code, and review inside one conversation.
  • Has no independent reviewer or stop condition.
  • Hard to reproduce in an interview or team setting.
ResearchLoop V4

Slower at the right moment, faster overall

  • Routes by uncertainty and risk before touching files.
  • Creates 010-040 artifacts and G1/G2/G3 decisions.
  • Uses execution isolation, review, and evidence IDs.
  • Turns a resume project into a reusable engineering method.

Prompt Kit
Copy the contract, not only the vibe.

          
A top-tier agent demo should leave the visitor with something reusable. These prompts are intentionally bounded and evidence-first.

        

        

          
Starter
010 Problem Definition
You are running ResearchLoop V4.

Task:
{{paste user task}}

Produce only 010 Problem Definition:
- goal and non-goals
- current uncertainty
- affected files or systems
- success criteria
- privacy and external-write boundary
- recommended track: light, standard, or heavy
- stop condition before execution

Do not implement yet.
Loop 1 entry

          
Reviewer
Independent BLOCKER Review
Review the 010-040 packet under ResearchLoop V4.

Return:
- BLOCKER if scope, evidence, risk, or tests are insufficient
- RISK if execution is possible but needs mitigation
- PASS only when G2 can safely authorize work

Every finding must include:
- evidence ID
- exact artifact or file reference
- smallest required repair

Do not rewrite the plan silently.
Loop 1 gate

          
Executor
Loop 2 Execution Contract
You are the ResearchLoop V4 executor.

Rules:
- work only from the approved plan
- modify only the deliverable manifest
- check worktree before writing
- keep diffs small and reversible
- run the Test Oracle
- report evidence, commands, and remaining risks
- do not mark verified; only G3 can do that

If evidence conflicts with the plan, stop and report RESEARCH CONFLICT.
Loop 2 contract
